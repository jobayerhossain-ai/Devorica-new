# DEVORICA BUTTON SYSTEM

Version: 1.0

Status: Core Component Documentation

---

# PURPOSE

Buttons are the most important interactive element in the interface.

Every button should communicate confidence before interaction.

Users should feel safe clicking it.

A premium button never asks for attention.

It naturally attracts it.

---

# DESIGN PHILOSOPHY

A button is not a colored rectangle.

A button is an invitation.

Every button should feel

• Human
• Premium
• Confident
• Calm
• Comfortable
• Clickable
• Intentional

Never create decorative buttons.

Create meaningful buttons.

---

# BUTTON PRINCIPLES

A great button should answer

Can I trust this?

↓

Should I click this?

↓

What happens after clicking?

The answer should always be obvious.

---

# VISUAL DESIGN

Buttons should have

Soft edges

Comfortable padding

Balanced typography

Subtle shadows

Thin borders when needed

No unnecessary gradients

No glossy effects

No bevels

---

# HOVER BEHAVIOR

Hover should feel alive.

Never dramatic.

Recommended

Slight Lift

↓

Tiny Scale

↓

Border Refinement

↓

Shadow Softening

↓

Arrow Translation

Animation Duration

180–220ms

---

# ACTIVE STATE

Button should compress slightly.

Never bounce.

Feedback should be immediate.

---

# FOCUS STATE

Visible.

Accessible.

Elegant.

Avoid default browser outlines.

Replace them with accessible focus styles.

---

# DISABLED STATE

Reduce emphasis.

Never reduce readability.

Cursor should communicate non-interactivity.

---

# LOADING STATE

Replace text with loader.

Maintain button width.

Never resize during loading.

Avoid layout shift.

---

# SUCCESS STATE

Only use when immediate confirmation is needed.

Example

Form Submitted

Project Created

Payment Successful

---

# ERROR STATE

Clearly communicate failure.

Do not rely on color alone.

Always include readable text.

---

---

# AI REVIEW CHECKLIST

Before approving a button ask

✓ Does it clearly communicate its purpose?

✓ Is it visually balanced?

✓ Is it easy to tap?

✓ Is it consistent with the design system?

✓ Is the primary action obvious?

✓ Is the typography readable?

✓ Are the hover and focus states meaningful?

✓ Does it feel handcrafted instead of generated?

✓ Would this fit naturally in an Apple, Linear, or Vercel-quality interface without copying them?

If any answer is NO,

redesign the button before shipping it.
